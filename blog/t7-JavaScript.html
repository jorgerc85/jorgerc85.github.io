JavaScript - Ruby classes vs. JavaScript constructor functions

November 2nd, 2014 by Jorge Rubio C.

Learning a new programming language has been a very interesting experience, learning a second programming language has been an even more interesting one, just like learning Spanish as my native language and then learning English as a second one.

Learning one language is similar than learning a second one but to me there is one key difference, being able to compare. Learning JavaScript has been an interesting experience because I've been able to use my knowledge of Ruby as a reference point to understand the logic and syntax of JavaScript.

As an example of this, I will dedicate this blog post to compare Ruby classes and JavaScript constructor functions. I have already covered Ruby classes in a previous post here, so I will focus on creating a parallel example to that one here to compare both approaches.

JavaScript constructor functions allow you to create new functions/objects that inherit characteristics from the original one much like Ruby classes do but with certain, many differences. Let's see them in action:

var Cars = function Cars() {
  // ...
}

var newCar = new Cars();

Using the '.prototype' function, you will be able to set properties, functions/methods and other objects to be inherited from the constructor function (the original 'Cars()' function) to all of the newly created functions based on it. This way, they will all share a similar set of properties and functions when created.

Cars.prototype.doors = 2;
var newCar = new Cars();

newCar.doors;  // --> 2

We can use this same pattern to inherit methods/functions. The only difference is that instead of assigning a value to a property, like in the example above, we assign a function. Like This:

Cars.prototype.drive = function drive() { return "You have arrived to your destination!"};
var newCar = new Cars();

newCar.drive;  // --> "You have arrived to your destination!"

Something really cool about the way JS operates is that you can both change properties of the constructor function afterwards and see those changes reflected on all of its child functions, and that you can also override properties for specific child functions is we want to.

// Change made after newCar's creation.
Cars.prototype.doors = 4;

newCar.doors;  // --> 4

// Override!
newCar.doors = 2;
newCar.doors;  // --> 2

var secondCar = new Cars();
secondCar.doors;  // --> 4

But what if I want to give my cars a distinguishing property in the moment of their creation?

var Cars = function Cars(brand) {
  this.brand = brand;  // 'this' in JS is "like" 'self' is in Ruby.
}

var thirdCar = new Cars("Nissan");

thirdCar.brand;  // --> "Nissan"

The approach JS has to "classes" is very different from Ruby's approach. It allows for certain similar capabilities but it also lacks part of the structure that Ruby has in place to define classes. This can be good or bad depending on what you're trying to do.

Now that you've read about both approaches, which one do you prefer? More class structure like in Ruby? or the simple and flexible constructor functions of JS?
