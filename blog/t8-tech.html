TDD: If it's worth building, it's worth testing

November 9th, 2014 by Jorge Rubio C.

Test-Driven Development (TDD) is a process used in software development that combines Test-First Development (TFD) techniques and refactoring to approach the development process in a much more efficient manner.

    Test-Driven Development = Test-First Development + Refactoring

The Test-First Development approach states that you should write test for your code before writing any functional code to your program. This tests should fail initially and the functional code written should aim to pass them. On the other side, refactoring is a programming technique that aims to improve code by making it easier to understand and modify by making small changes that improve your design. Refactoring doesn't add new functionality, it just edits your code to make it better.

Refactoring can be useful both after a new feature has been added and before adding a new feature to older code. The use of refactoring before adding new code can help making the new features easier to implement since the code already in existence could be easier to modify if we edit it first.

The use of both of these techniques combined should allow for an incremental expansion of the code in a program. First, you add a test (this should be a small and simple test). Second, you create the code necessary to pass the test. As a third step, you refactor your code and then go back to the beginning to add a new test. This loop should continue until the initial functionality that was expected is achieved.

                     ------> Add test <------
                     |           |          | [Pass]
                     |           v          |
                     |       Run the tests --
                     |           | [Fail]
                     |           v
               [Pass]|       Make small changes <--
         [Development|           |                | [Fail]
           Continues]|           v                |
                     ------- Run the tests --------
                                 | [Pass]
                                 v [Development stops]

This approach should help you tackling errors as early as possible in the development process and should allow you to find the piece of code that generates the error faster. It is a lot easier to figure out what is wrong with a small piece of code than thousands of lines of it.

There are many ways in which TDD can be applied, scaled and used in other development processes, like database development for example. Although it is not a rule to use it, it can save you many hours of work and can improve the quality of your code greatly.

As a summary, TDD is both a great tool for testing your code at an early stage of development and a great way of producing better quality code.
